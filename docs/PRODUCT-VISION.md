# ZENSHIN CHART — プロダクトビジョン

## 何のためのツールか

ZENSHIN CHARTは、ロバート・フリッツの「緊張構造（Structural Tension）」理論に基づく思考・実行ツールである。

Vision（理想の状態）とReality（今の現実）のギャップを明確にし、そのTension（緊張）を維持しながら、Actionを通じて前進し続けることを支援する。

**核心**: 緊張構造が張られている限り、人は創造に向かって動き続ける。問題は、この緊張が気づかないうちに緩んでしまうことにある。

---

## 提供価値のサイクル

ZENSHIN CHARTの価値は、以下の4つのフェーズで構成される使用サイクルの中で発揮される。
```
   ① 書く（言語化）
       ↓
   ② 観る（俯瞰）
       ↓
   ③ 気づく（検知）
       ↓
   ④ 動く（前進）
       ↓
   ① に戻る
```

### ① 書く — エディターで構造を言語化する

**体験**: Vision / Reality / Tension / Action を自分の言葉で書き出す

- 描き心地が良いエディターであることが最も重要
- 思考の流れを止めない入力体験
- 構造が自然と整理される UI

**これが磨かれていなければ、そもそも書かない。書かなければ何も始まらない。**

### ② 観る — ビューアーで構造を俯瞰する

**体験**: 書いた内容をカンバン / ツリー / 俯瞰ビューで多角的に観察する

- 同じデータを異なる視点で見ることで、構造の全体像が浮かび上がる
- アクションの進捗、停滞、偏りが視覚的にわかる
- 「書いたものを眺める」行為自体が思考を深める

**しっかりと観察できるビューアーがなければ、書いたきりで終わる。**

### ③ 気づく — 変化と停滞を検知する

**体験**: ダッシュボード / 通知 / AI示唆で、自分では見落としていた兆候に気づく

- 更新が止まっているチャート
- 期限切れ・担当者未設定のアクション
- テンションの緩みの兆候（後述）

**①②が日常的に使われていなければ、③の通知は空虚なノイズになる。**

### ④ 動く — アクションを更新して前進する

**体験**: 気づきを元にチャートに戻り、Tension を再確認し、Action を更新する

- Slack / メール / サイト内通知から1クリックでチャートに戻れる
- 編集 → 保存 → 前進のサイクルが滑らかに回る

## βテストでの検証優先順位
```
最優先   ① 書く（エディター）  ← 描き心地が全ての起点
         ② 観る（ビューアー）  ← 書いたものを活かす体験
---
次フェーズ ③ 気づく（ダッシュボード / 通知）
         ④ 動く（Slack連携 / AIコーチング）
```

①②が検証されて初めて③④の意味が生まれる。βテスターが「書きたくなる」「眺めたくなる」状態を先に作る。

## テンションの緩みとAIコーチング

### 緊張構造の動態

フリッツの理論において、緊張構造は静的なものではない。
```
【初期】
  Vision ━━━━━━━━━━━ 強いTension ━━━━━━━━━━━ Reality
  （理想は明確、現実との差が大きい）

【Action後】
  Vision ━━━━━━━━━━━ Tension ━━━━━━ Reality'
  （現実が動いた。Tensionは少し緩む）

【進行中のリスク】
  Vision ━━━━ ? ━━━━ Reality''
  （現実の解像度が上がったことで、元のVisionが曖昧に感じ始める）
  （Tensionが緩み、創造が停滞する）
```

Actionを起こし、失敗と成功を重ねることで Reality は変化する。現実に対する解像度が上がる。

この過程で起きうること：
- **Visionが古くなる** — Realityが進んだのに、Visionが最初のまま
- **Tensionが曖昧になる** — 何が課題なのか分からなくなる
- **Actionが惰性になる** — やることはあるが、なぜやるのか忘れている

これが「テンションの緩み」であり、創造の停滞の正体である。

### AIの役割 — 問いを投げるコーチ

人間は俯瞰して意味づけする能力に長けている。新しいVisionを描き直し、Tensionを張り直すことができる。

AIは構造の変化を定量的に追跡し、パターンを検知する能力に長けている。人間が見落とす兆候を拾い上げることができる。

**AIコーチングの原則: 答えを出すのではなく、問いを投げる。**

| AIが検知すること | 投げかける問い（例） |
|----------------|-------------------|
| Realityが大きく更新されたが、Visionが変わっていない | 「Realityがだいぶ進みましたが、このVisionはまだ同じ表現で合っていますか？」 |
| Tensionに紐づくActionが全て完了したが、Tensionが閉じていない | 「このTensionのActionは全て完了しました。Tensionは解消されましたか？それとも新しいActionが必要ですか？」 |
| 長期間Actionが追加されていないTension | 「このTensionに最後にActionを追加してから30日経ちました。次の一手はありますか？」 |
| 定量的な指標がないVision | 「このVisionを達成できたかどうか、どうやって判断しますか？」 |
| Tensionに紐づいていないAction | 「このActionは『○○』のTensionに関連しそうですが、紐づけますか？」 |
| 同じエリアのVisionとRealityに変更がないまま90日経過 | 「このエリアの状況は3ヶ月前と変わっていないようです。何か変化はありましたか？」 |

### AIコーチングの設計思想

- 提案は「示唆」であり「指示」ではない
- ユーザーが無視しても問題ない（同じ提案を繰り返さない）
- 緊張構造の理論に基づいた問いであること
- 最終的な判断と意味づけは常に人間が行う

## 人間とAIの役割分担

| | 人間 | AI |
|---|------|-----|
| 得意なこと | 俯瞰して意味づけする。Visionを再定義する。新しいTensionを張り直す。文脈を読み、創造的な判断をする。 | 構造の変化を定量的に追跡する。抜け漏れを検知する。パターンから兆候を拾い上げる。全データを横断的に比較する。 |
| 苦手なこと | 自分の変化に気づきにくい。日常に埋もれてテンションの緩みを見過ごす。 | 意味を理解すること。何が本当に重要かを判断すること。 |
| ZENSHINでの役割 | 書く。観る。考える。決める。動く。 | 兆候を検知する。問いを投げる。構造の健全性をモニターする。 |

## 最大の課題 — チャートの鮮度問題

### 問題の本質

行動を起こしていく過程で、経験や考え方の変化、現実の変化をチャートに記述していくことは非常に億劫になる。

**なぜか**: 行動が主役であり、記述は「共有のための作業」だという認識を持ってしまうから。

### 解決の方向性

根本原因は **「記述コスト > 記述から得られる価値の実感」** になること。この不等式を逆転させる必要がある。

#### A. 記述コストを極限まで下げる

| アイデア | 概要 |
|---------|------|
| Slackからの自動更新提案 | Slackでの会話をAIが解析し更新を提案 |
| アクション完了時のReality更新促し | 完了時に「現実はどう変わりましたか？」と1行入力を促す |
| 週次の1問リフレクション | 「今週、最も変化したRealityは何ですか？」 |
| 音声入力 | 移動中にスマホで口頭でReality更新 |
| コメントからの昇格 | タイムラインのコメントを「Realityに反映」ボタンで昇格 |

#### A+. 外部サービスからRealityを自動集約する

Slack、Notion、ClickUp、GitHub等から情報を引っ張り、Realityの変化を自動検知・提案する。自動で書き換えるのではなく、提案として提示する。

#### B. 記述の価値を即座に実感させる

| アイデア | 概要 |
|---------|------|
| 更新時のTension変化可視化 | Realityを更新した瞬間にフィードバック |
| Before/After表示 | 30日前と今を並べて進歩を実感 |
| チーム通知 | 更新するとチームに通知 |
| AI洞察の即時生成 | 更新した瞬間にAIが示唆を返す |

#### C. 更新しないことのコストを可視化する

| アイデア | 概要 |
|---------|------|
| 鮮度インジケーター | 最終更新日と色のフェードアウト |
| 停滞アラート | AIコーチからの問いかけ |
| 進捗レポートでの空白 | 週次ダイジェストで「更新なし」を可視化 |

### 設計原則

- **義務感で書かせない** — 「書くと良いことがある」
- **最小の入力で最大の更新** — 1行、1クリック、1音声
- **行動の文脈の中で更新が起きる** — チャートを開くための別の時間を作らせない
- **組織では特に重要** — 個人は頭の中で補完できるが、組織ではチャートが唯一の共有媒体

---

## まとめ

ZENSHIN CHARTが実現したいのは、**緊張構造を張り続けるための環境**である。

1. **書きやすいエディター**で構造を言語化し
2. **見やすいビューアー**で俯瞰し
3. **AIコーチ**がテンションの緩みに気づかせ
4. **通知**がチャートに戻るきっかけを作り
5. **人間**が意味づけし直して、前進を続ける

このサイクルが回り続ける限り、創造は前に進む。
